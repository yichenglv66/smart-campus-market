# 数据库课程作业说明（Smart Campus Market）

本文档用于说明作业中的安全控制、并发问题、事务处理和故障恢复思路。

## 1. 如何防止普通用户删除数据

### 已在代码中实现
- 业务层限制：已售出商品不能删除。
  - 在 `POST /items/{item_id}/delete` 中，如果 `item.status == 1`，直接返回失败提示，不执行删除。
- 业务层限制：不存在的商品不能删除。

### 概念性说明（建议在数据库层补充）
- 使用 MySQL 权限控制：
  - 管理员账号：允许 `INSERT/UPDATE/DELETE/SELECT`
  - 普通账号：只允许 `SELECT`（或只允许部分表的 `SELECT`）
- 应用连接数据库时，前台读操作可使用只读账号，管理操作使用管理员账号。

> 结论：当前项目已做“应用层删除限制”；更稳妥的“数据库账号权限限制”属于部署配置，代码中尚未完整落地。

## 2. 如何限制普通用户只能查询数据

### 已在代码中实现
- 当前页面路由主要是查询页面（首页、用户列表、商品列表、订单列表、报表页），都使用 SQLAlchemy 查询并渲染。
- 关键写操作（新增/改价/删除/购买）是单独路由，不会在普通只读页面自动触发。

### 概念性说明（建议在数据库层补充）
- 为普通用户创建只读数据库账号：仅授予 `SELECT`。
- 后端为“普通访问”与“管理访问”区分不同数据库连接（课程作业可只在文档说明，不强制多连接实现）。

> 结论：当前代码做了“页面与操作路由分离”；真正“只能查不能改”应依赖数据库权限策略。

## 3. 两个用户同时购买同一商品时会出现什么问题

### 可能问题（并发冲突）
- 两个请求几乎同时看到商品状态为未售出（`status=0`）。
- 如果不加控制，可能出现：
  1. 重复生成两条订单；
  2. 商品状态更新混乱；
  3. 页面显示与真实库存不一致。

## 4. 如何通过事务或加锁解决

### 已在代码中实现
购买接口 `POST /items/{item_id}/purchase` 使用了以下机制：
1. 使用事务：`with db.begin()`，保证“插入订单 + 修改商品状态”是一个原子操作。
2. 行级锁：查询商品时使用 `with_for_update()`，锁住该商品行，避免并发下多个请求同时通过校验。
3. 状态检查：如果 `item.status == 1`，直接拒绝购买。
4. 数据库唯一约束兜底：`orders.item_id` 唯一，保证同一商品最多一条订单。
5. 异常回滚：`IntegrityError` 或其他异常会回滚事务并提示失败。

> 结论：当前代码已经实现了作业要求中的事务与并发控制核心方案。

## 5. 如果系统崩溃，如何恢复订单数据

### 已在代码中实现
- 事务提交前崩溃：`with db.begin()` 事务不会提交，数据库会自动回滚未完成变更。
- 事务提交后崩溃：已提交数据由数据库持久化保存，重启服务后仍可查询。

### 概念性说明（建议在运维层补充）
- 定期备份：每天导出数据库（`mysqldump`）。
- 开启 MySQL binlog（可选，课程中可作为附加说明）用于时间点恢复。
- 恢复步骤（简化版）：
  1. 恢复最近一次全量备份；
  2. 如有 binlog，再回放到故障前时间点；
  3. 重启应用并校验订单表与商品状态一致性。

> 结论：当前项目依赖数据库事务保证一致性；完整灾备恢复属于部署与运维方案，文档中给出课程级可行做法。
